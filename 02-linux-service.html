TCP 和 UDP 协议

[图：TCP/UDP 协议对比图]

TCP 协议

  • 面向连接：在传输数据的时候，要建立客户端和服务器的连接，才能传输数据
  • 可靠的通信：TCP 传输数据的时候，会基于内部的各种机制保证数据传输到目的地
  • 基于字节流：TCP 传输的是基于字节传输，易于对数据的拆分和合并发送

UDP 协议

  • 无连接：只需要知道目的地 IP 和端口号就可以发送数据，无需建立连接
  • 不可靠的通信：没有一系列机制应对传输时丢包的问题
  • 面向数据包发送：应用层给 UDP 什么报文，UDP 就会发送什么
  • 适用范围：由于 UDP 是不属于连接性协议，所以消耗资源比较小，处理速度优的特点，因此适用于视频、音频通话的时候

TCP 与 UDP 对比

TCPUDP
连接有连接无连接
可靠可靠不可靠
发送面向字节流面向数据包
UDP 传输的时候消耗比 TCP 小，速度更快

TCP 三次握手

[图：TCP 三次握手示意图]

  • 客户端发送一个 SYN（同步标识包）给服务端
  • 服务器接收到 SYN 包后，回传一个 SYN/ACK 的数据包传递确认信息，表示我收到了
  • 客户端再回传一个 ACK 标志的数据包，表示我知道，握手结束

TCP 四次挥手

[图：TCP 四次挥手示意图]

  • 客户端发送一个 FIN，用来关闭客户端到服务端的数据发送
  • 服务端在收到 FIN 的时候，会返回一个 ACK 给客户端
  • 服务端发送一个 FIN 给客户端，用来关闭服务端到客户端的数据发送
  • 客户端收到 FIN 后会返回一个 ACK 给服务端，确认断开

Linux 服务 - FTP（vsftpd 配置 / 主动被动模式）

▼

1.1 什么是 FTP

  • 文件传输协议：适用于网络上进行文件传输的一套标准协议，它是工作在 OSI 七层应用层，TCP/IP 的五层应用层

1.2 FTP 的工作原理

FTP 是采用互联网标准文件传输协议，给用户提供一组管理计算机之间文件传输的应用程序。

FTP 是基于 C/S 模型（客户端、服务器）设计的，在客户端和 ftp 服务器之间建立连接。

1.3 传输方式

1.3.1 ASCLL

  • 假设用户拷贝的文件包含了简单 ASCII 码文本，就算远端的计算机不是 Linux 系统，当文件传输的时候，ftp 会自动调整文件的内容，方便把文件解析为另一台电脑可以存储的文本格式
  • ASCII 更通用，效率没有二进制高

1.3.2 二进制

  • 在二进制传输的时候，保存文件的位序，以便原始和拷贝的逐位一一对应，即使目的机器上包含位序的文件是没有任何意义，所以基本是不能互通

五、FTP 工作模式

5.1 主动模式

工作原理：

  • FTP 客户端连接 FTP 服务器开放的 21 端口，发送用户名、密码登录
  • 客户端随机开放一个端口（1024 以上），把随机端口给到服务器
  • 服务器接收到随机端口后，会使用 20 号端口连接客户端开放的随机端口

[图：FTP 主动模式示意图]

5.2 被动模式

工作原理：

  • FTP 客户端连接 FTP 服务器开放的 21 端口，发送用户名、密码登录
  • 客户端发出 PASV 命令，告知服务器被动模式，控制连接建立成功
  • 连接成功之后，服务器会生成一个随机端口，将这个端口告知给客户端
  • 客户端主动连接服务器随机端口，建立数据连接

[图：FTP 被动模式示意图]

5.3 如何选择

  • 如果你们是在一个局域网，或者你在一个计算机上，可以使用主动模式
  • 如果是在公网，比如阿里云服务器建议使用被动模式

Linux - DHCP（原理 / 部署）

▼

1.1 什么是 DHCP

DHCP（动态主机配置协议），是一个局域网的网络协议，使用 UDP 协议。

作用：给内外主机自动分配 IP 地址，给用户或者内部的网络管理员作为管理所有计算机的手段。

DHCP 是 C/S 模型：DHCP 客户端和服务器要保持通信，DHCP 是基于 UDP 协议，双方都有可能主动向对方发起通信，因此双方都需要监听端口：

  • server 端：67/UDP
  • client 端：68/UDP

1.2 DHCP 的优势

  • 减少管理员工作量
  • 避免输入错误的可能性
  • 避免 IP 地址冲突
  • 更改 IP 网段的时候，不需要手动重新分配每一个 IP 地址

1.3 DHCP 的分配方式

  • 手动分配：由 DHCP 服务器管理指定 IP 地址
  • 自动分配：分配 IP 地址之后，永久使用
  • 动态分配：使用完成之后会自动释放 IP，供其他用户使用

1.4 DHCP 的工作原理

[图：DHCP 工作原理示意图]

  • 客户端请求 IP 地址：客户端启动之后，本身是没有 IP 地址，通过广播的方式找 dhcp 服务器
  • 服务器响应：服务器在收到请求，会在自己的 IP 地址池查找是否有合法，如果查找到之后会给 IP 做个记号加入到 dhcp offer 内发送给客户端
  • 客户机选择 IP 地址
  • 服务器确认租约

PXE

1.1 什么是 pxe

  • pxe（预启动执行环境）
  • pxe 是 intel 公司推出的一款通过网络来引导操作系统的协议，广泛应用于网吧无盘系统
  • 只需要把无盘服务器维护好，就能保证网络中无盘工作站（电脑）的系统正常运行
  • PXE 服务器需要支持软件：DHCP、tftp、syslinux（pxe 引导的程序文件） 文件共享（samba、nfs）

1.2 PXE 服务器和客户机的工作流程

[图：PXE 工作流程示意图]

  1. pxe 客户机发送 DHCP 请求，向服务器申请 IP 地址
  2. DHCP 服务器响应 PXE 客户机的请求，自动从 IP 地址池分配 IP 地址给客户机，并告知 TFTP 服务器 IP 地址和 pxe 引导程序 pxelinux.0
  3. PXE 客户机向 tftp 服务器发起获取 pxelinux.0 引导文件
  4. TFTP 服务器响应 PXE 客户机的请求，将其共享的 pxelinux.0 引导程序给 pxe 客户机
  5. pxe 客户机通过网络来启动安装界面
  6. pxe 客户机向 samba 发起获取操作系统文件请求
  7. samba（文件共享服务器）响应 pxe 客户机的请求，将共享文件系统传输给 pxe 客户机
  8. pxe 客户机进行安装提示相对，客户需要手动解决安装
实际安装的时候，可以采用 kickstart 应答文件，来进行无人值守安装。

Cobbler

2.1 什么是 cobbler

Cobbler 是一个快速网络安装 Linux 的服务，现在也可以通过自定义脚本网络安装 windows。利用一些简单的命令就可以完成 pxe 网络安装环境的配置，同时还可以管理 DHCP、DFS、TFTP、rsync 以及 yum 仓库。

2.2 cobbler 的作用

  • 服务器上架后，可以手动选择需要安装的操作系统
  • 服务器上架后，可以根据需求安装配置操作系统
  • 系统安装后，可以执行自定义脚本，安装系统基础软件初始化

2.3 kickstart 是什么

  • kickstart 是 redhat（红帽）开发的一个工具，可以理解为：一个自动安装应答配置管理程序
  • kickstart 安装选项，包含：语言、防火墙、密码、网络分区等等配置

1.2.2 Redis（6379）

  • 存储数据，但是和mysql不一样

  • 存储选择：数据最终存储mysql数据库（硬盘永久数据）；如果希望数据查询功能比较强大，使用关系型mysql；如果希望性能高效，使用redis利用内存数据；如果不希望使用关系查询，可以使用hash结构redis
