1.1 MyISAM

Mysql5.5版本之前的数据库的默认引擎
    特点：
        1.不支持事务机制
        2.支持表级别的锁定
        3.数据单独写入或是读取的时候，访问速度比较快，占用系统资源少
        4.读写的相互阻塞，读的时候不能写，写的时候不能读
        5.只能缓存索引，不能缓存数据
        6.每个使用myisam引擎的表，同时会创建三个文件
            *.frm    存储表结构信息
            *.MYD    存放表的数据
            *.MYI    存放索引信息


适用场景：
    MyISAM适合用于数据库主要以读或者写为主的操作
    只有少量的更新或删除
    对事务完整性和并发要求不高的生产环境
1.2 InnoDB
Mysql5.5版本之后的数据库的默认引擎
    特点：
        1.支持事务机制
        2.支持行级锁定，但是在扫描的时候，显示的是表级锁定
        3.支持外键，来保证数据的完整性和正确性
        4.提高缓存：索引和数据都可以缓存       
        5.读写分离，相互是不阻塞
        6.Innodb引擎的数据库，会创建两个文件
            *.frm    存放表结构
            *.ibd    存放索引和数据

适用场景：
    适合数据会频繁更新以及操作的场景，且对事务有一点要求

  • 事务机制特性（ACID）

    • A-原子性

    • C-一致性

    • I-隔离性

    • D-持久性

三、数据库用户管理

数据库用户：
    root：最大的管理员，用户所有权限（负责运维、开发经理）


开发或者项目
    由负责人创建用户，分配给工作人员使用，设置相关权限
创建普通用户
#语法格式
#密码：8位以上包含英文大小、数字、特殊符号
#这个用户，可以连接数据库，但是只能连接并不能看到相关数据
create user '用户名'@'localhost' identified by '密码'；

mysql> create user 'aaa'@'localhost' identified by 'Bbc123456!';

  • 预览用户权限

四、数据库备份

4.1 为什么要备份？
备份的目的：解决数据库的灾难恢复，实现容灾
数据丢失原因：
    开发人员有没有出来号的bug
    负责人和设备：操作失误、天灾人户、设备老化、黑客攻击……
4.2 备份的方式

  • 物理备份

    • 把操作系统的物理文件备份

      • 冷备份：把数据库关停进行备份：tar

      • 热备份：让数据库正常运行进行备份：backup\hotcopy

      • 温备份：数据库锁定的时候进行备份：mysqldump

  • 逻辑备份

    • 完全备份：全部

    • 差异备份：备份以上次备份不同数据

    • 增量备份：对比上一次备份后新增的内容，进行备份

4.3 温备份 mysqldump
利用mysql自带的备份攻击，把指定的库、指定表导出生产一个sql脚本
实际上就是一个记事本，后缀名.sql
把表格、数据等信息生成执行语句，下次需要恢复数据的时候，全部执行一次

#导出abc库到/opt/abc.sql
[root@localhost ~]# mysqldump -uroot -p --databases abc > /opt/abc.sql
Enter password:

#导出abc和cateye1两个库到/opt/abc-cateye1.sql

4.4 冷备份
把指定数据文件打包成.tar.gz

二、 数据类型

数据库：帮助实现数据持久化存储功能
数据在存储的时候，需要刻录硬盘，同时还要支持在内存上操作
不同的数据，格式不一样
目的：是为了最大化节约系统资源，提高同类型之间的数据运算效率

=================================================================

整数型
    tinyint    整数：0-255
    samllint    整数：-32768~32767
    int        整数：10位以内
    bigint     整数：10位以上

浮点型
    float（m，d）    非整数：单精度浮点
                m代表整数，d代表小数位数：比如：如果设置长度为10，m写3，d纪要写7
    double（m,d）    双精度浮点数
    decimal（m,d）   M最大为65，D最大30

字符类
    char（m）    固定长度，内容不足，自动填充空格    
                缺点：浪费磁盘空间
                优点：查询操作速度快
    varchar（m） 可变长度，内容不足，按照实际大小
                 优点：节省磁盘空间
                 缺点：查询操作速度慢
     text        默认时varcha（max）大小，可变长度，适合用于大段的内容

四、索引

索引的特点

    帮助mysql快速查询更新数据库表内的数据

    无需对整个 表进行匹配扫描，通过物理地址找到相应的数据

    针对某些字段继续建立索引，并非对所有的字段都建立索引

索引的优点

    当表内的数据量很大或者查询的涉及到了很多个表，使用索引，可以提高查询速率，降低数据库的IO成本

    通过索引对数据进行排序，可以降低数据库的排序成本，降低了CPU的消耗

    通过去建立唯一索引，来保证数据库种每一行数据的唯一性

    还能加快表与表之间的连接

索引的缺点

    建立索引会增加数据库的存储空间

    当对表的数据进行insert、update、delete的时候，索引也要动态的维护，因为更新数据库表的时候，mysql不仅仅要保存数据，还需要保存索引文件

索引的分类

普通索引：没有唯一性，一个表内可以有多个普通索引

唯一索引：值必须是唯一的，可以写空值，一个表内可以创建多个唯一索引

主键索引：一个特殊的唯一索引，主键索引要求每个值都是唯一的，不允许有空值，且一个表只能由一个主键索引

组合索引：不具备唯一性，一个表内可以有多个

创建索引的原则

    表中的数据超过300条应该有索引

    表中唯一性比较差的字段不建议创建索引

    数据表中更新太频繁的字段不适合做索引

    经常出现在where字句中的字段，特别是大表的字段，应该建立索引

    要保证表内某一个字段的唯一性，可以建立索引

索引的分类
主键索引:
    它和主键primary key绑定的，一张表只能创建一个
    它自带：非空+唯一+自动升序
唯一索引：
    它单独创建， unique，数据必须是唯一的，允许为空，一个表内可以有多个
普通索引：
    它单独创建，index，只是和表格数据对应，没有任何特效
组合索引：
    它是多个列同时放在一起的索引，只是和表格数据对应

一、主从复制的工作原理

1.首先主库需要开启二进制日志，需要授权给从库访问 2.从库会先开启一个IO线程，利用IO线程和主库建立连接，并且读取主库bin.log日志内容，将读取到的日志内容写入到relay.log；如果此时主库和从库数据一致，此时IO线程会休眠，直到主库产生新的内容才会同步 3.从库会再次一个sql线程，sql线程会读取relay.log日志内的内容并且在从库执行一般，从而实现主从复制

二、主从复制的类型

类型

    1.基于语句复制（默认）

    2.基于行的复制

    3.混合类型的复制

操作

    1.异步复制（默认）

        数据库的主库不会把bin日志内容主动推送给每一个从库

        执行完毕之后直接返回结果给客户端

        它是不管从库是否已经接收并且处理好

        效率很高，但是容易出现问题

            1.主库操作没有推送到仓库

            2.会出现两个库数据不一样

    2.同步复制

        当主库执行完成一个事务后，会等待所有的从库复制且执行成功才会返回信息给客户端

        效率和性能都会收到影响

    3.半同步

        当主库执行完一个事务后，会等待至少一个从库写入到relay日志内，然后返回结果给客户端
