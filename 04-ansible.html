Ansible 自动化运维

一、ansible 入门

1.1 什么是ansible

  • Ansible是IT自动化管理工具，集成了很多模块以及功能组件，可以通过一个命令完成一系列，减少工作的复杂性

  • 工作内容：主要是集中在管理、安装、部署等等

    • 比如：10个Linux服务器，现在需要安装mysql服务，可以利用ansible。它是可以他是操作10台机器
Ansible和Jenkins没有可比性


    Jenkins：自动化部署，它是Java开发的，需要JDK环境
            主要用于部署java语言开发的项目，集成内置插件
            可以连接gitee、gitlab、gitub，拉取、推送
            利用maven做打包项目、发布 
            可以同时推送到多台设备-集群，同时部署项目


    Ansible：自动化运维，它是python语言开发，需要python环境
            主要用于运维工作，管理所有的互联网服务器
            可以实现同时执行：命令、安装、部署等等
            可以在一个文件，管理所有的设备，还可以进行针对性的处理
1.2 ansible 主要功能

  • 批量执行远程命令，可以对N台主机同时运行命令

  • 批量配置软件服务，可以进行自动化配置和管理服务器

  • 实现软件开发，Jumserver底层就是ansible

  • 编排改进IT任务，Ansible有一个:playbook，属于ansible自己的编程语言，支持自定义开发

1.3 Ansible特点

  • 容易学习，无需代理

  • 操作灵活，模块很多，功能强大，还可以使用playbook进行编程辅助

  • 安全性高，使用ssh协议进行通信的

  • 移植性很高，可以把写好Playbook拷贝到任何服务器运行

五、Ansible ad-hoc

5.1 什么是ad-hoc
  • ad-hoc指的是临时指令：临时命令，执行完毕后就结束了，不会保存
主命令 主机名称 指定模块 模块 动作 命令
ansible all（所有主机）\主机组\IP  自定义 -m（固定参数） ping -a df -h

上述标黄的内容，都是可以进行修改

主机名称：可以自定义指定单个主机、主机组或者是所有被动端

模块：各种模块功能，可以自己更改

命令：在被控端执行linux命令

5.2 ad-hoc执行过程

  • 加载配置文件：/etc/ansible/ansible.cfg

  • 查找对应的主机配置文件

  • 加载对应的功能模块文件，比如：ping

  • 通过ansible将模块生成对应的文件：.py（python格式的文件）；把文件传输给被控端：$HOME/.ansible/tmp/ansible-tmp-number/**.py

  • 然后给.py文件添加执行权限

  • 然后执行，返回结果

  • 删除.py的临时文件

  • 退出

5.3 ad-hoc的执行状态

192.168.10.130 | CHANGED | rc=0 >>

192.168.10.129 | CHANGED | rc=0 >>

  • 绿色：代表被控主机没有发送改变

  • 黄色：代表被控主机发生了变动（宇当前执行的时候不同）

  • 红色：代表被控主机发生了故障

6.4 yum 模块
-m yum
    提供管理各个被控端的系统的软件包
指令参数 选项 说明
name nginx、httpd…… 指定安装软件的url
state prnsent、installed、latest、absent、removed prnsent 默认安装；installed 安装；latest 安装最新版本；absent、removed表示卸载。允许多个软件一次性安装，使用逗号隔开就可以
enablerepo epel、base 指定从那些仓库获取软件，密钥等等
disablerepo epel、base 禁止从哪些仓库获取软件等等
exclude httpd 排除某些指定包
download_only yes，no 只下载不安装

一、Playbook介绍

1.1 什么是playbook
palybook也是yml语法编写的文本文件，主要是由play和task组成


    play：    主要是定义主机、主机组
                用hosts:定义
    task：    主要定义对主机或者主机组的任务，可以是一个，也可以是很多个
                用tasks:定义
总结：
    在编写playbook的时候，里面由一个或者是多个play组成；
    每一个paly可以包含一个或者多个task；
1.2 playbook和ad-hoc的区别

playbook是ad-hoc的一种编排方式

playbook是可以持续运行的，但ad-hoc只能临时运行一次

playbook适合复杂的任务，ad-hoc适合执行简单的任务

playbook能控制任务先后执行顺序

1.3 playbook的编写要求
Paybook是yml语言编写，结构要清晰，可读性强，很多语言都是需要这种语法定义
这个yml语言的要求非常严格，空格也是有明确要求的


================================================================
要求
    1.文件中，禁止使用Tab进行缩进，如果使用编辑软件，需要修改配置
        将tab缩进，替换空格：必须是双数：2、4、6、8……
    2. 文件编码格式，必须和操作系统匹配，默认utf-8
        如果使用vi或者vim，它自动和系统匹配


缩进：
    yml和yaml使用固定缩进风格表示内容层级，每个缩进由双数的空格对应
    通常都是2个空格代表一层


冒号：
    除了以冒号结尾的，所有冒号后面，必须有一个空格,举例： （name: zhangsan）


减号：
    减号代表项，减号后面必须有空格，举例： - name

  • 使用vi或者vim编写playbook（最推荐的，因为他会自己匹配编码格式）

  • 如果要在windows系统内编辑playbook，保存时候一定要使用utf-8编码格式

  • 如果使用moba提供的文本工具，也要调整编码格式

二、Ansible variables

2.1 什么是变量
一个动态获取数据的对象，在ansible中，提供指定名称定义，然后预存数据
当指令或者yml操作的时候，可以调用里面的数据
2.2 变量的定义方式
在Ansible内，定义变量有三种方式
    - 通过命令行传递变量参数定义
    - 在playbook文件定义变量
        - 通过vars：    定义变量
        - 通过vars_files 定义变量
    - 通过Inventory在主机组或者单个主机中设置变量
        - 通过hosts_vars:  对主机进行定义
        - 通过group_vars:  对主机组进行定义
2.3 优先级
playbook内可以定义多种变量，但是传递和执行的时候是有优先级的关系的
2.4 在playbook中定义变量
2.4.1 vars方式

  • 在playbook文件中，头部通过vars：关键字 进行变量的定义

  • 所有变量的数据，只能在当前文件使用

  • 变量名是可以自定义，必须符合标识符命名规则

    • 首字母必须是：字母或者下划线

    • 其余部分：可以是字母、数字、下划线

4.1 循环的概念
loop:循环属于当前任务的一个功能，归属于- name下面
循环中，提供一个个数据的项，每一项都要由- 定义
它会一项项重复执行当前的任务，没执行一次就会提取一项值，交给变量{{ item }}
注意：当前循环只对当前的任务有效的，任务执行的时相同功能
4.2 循环启动服务

    - name: "启动firewalld"

五、Handlers和Notify

  • Handlers是一个触发器，同时它也是一个tasks任务

  • Handlers不能直接运行，需要被其他task通知才能运行

  • Notify就是通知，他告诉handlers要执行操作

#我想要被控端运行的httpd服务的端口号是7777
#主控端下发配置文件到客户端
#在下发文件后，自动重启httpd

- hosts: webservers
  tasks:
    - name: "安装http服务"
      yum: name=httpd state=latest

    - name: "下发配置文件"
      copy: src=/opt/httpd.conf dest=/etc/httpd/conf/httpd.conf
      notify:
        - restart_httpd

    - name: "启动httpd服务"
      service: name=httpd state=started

  handlers:
    - name: restart_httpd
      service: name=httpd state=restarted

执行顺序：

    tasks：定义任务列表，任务执行顺序是一个个执行的

    handlers： 定义的触发任务，默认是不执行的

    handlers执行条件：

        当tasks列表中，某一个任务定义notify通知handlers，需要明确指明触发器的名称，最个时候，handlers中触发任务，才会在task任务执行完毕后再执行

handlers的注意事项

  • 无论有多少task通知相同的handlers，handlers只会在所有的task任务执行完毕后，执行一次

  • 只有task发生了改变，才会停止handlers，没有改变是不通知的

  • 不能使用handlers代替task，handlers默认不执行

八、Playbook异常处理

  • 在playbook执行过程中，会遇到各种错误，由于playbook遇到错误之后，不会再执行后续的任务，不方便调试
8.1 执行失败-继续执行后续任务

  • ignore_errors: 他是任务的属性，分配给任务，属于- name

    • yes：忽略跳过；no：终止（默认）
[root@localhost opt]# cat an-16.yml
- hosts: webservers
  tasks:
    - name: "错误任务"
      shell: asbdasdasdsd
      ignore_errors: yes


    - name: "创建文件"
      file: path=/opt/123.txt state=touch
8.2 task任务执行失败后，强制调用handlers

  • 通常情况下，task任务执行失败后是中断。

  • 可以强制调用handlers

    • force_handlers: yes

      • 默认是no，yes强制开启

      • force_handlers负责管理任务，他定义再- hosts下

- hosts: webservers
  force_handlers: yes
  tasks:
    - name: "创建文件"
      file: path=/opt/456.txt state=touch
      notify:
        - restart_httpd
    - name: "安装服务"
      yum: name=wwh state=latest
  handlers:
    - name: restart_httpd
      service: name=httpd state=restarted
8.3 控制task任务状态报告

  • 很多任务在执行完毕后，本身数据没有发生改变，但是每次执行任务，都会返回黄色

  • changed_when: false 不让他做提醒

    • 针对某一个任务，属于- name

    • 这个属性可以自定义

[root@localhost opt]# cat an-17.yml
- hosts: webservers
  force_handlers: yes
  tasks:
    - name: "创建文件"
      file: path=/opt/456.txt state=touch
      notify:
        - restart_httpd
      changed_when: false

    - name: "安装服务"
      yum: name=wwh state=latest
      ignore_errors: yes
  handlers:
    - name: restart_httpd
      service: name=httpd state=restarted

  • 正常情况

    • task任务执行后，被控端发生改变，notify通知handlers，然后执行handlers内的任务

  • changed_when控制任务状态报告

    • 它有一个检查的判断，来验证数据是否针对没有发生改变

    • 重复创建文件，数据本身没有发生改变，但是文件创建时间发生改变=被控端发生给便通知handlers

    • 利用 changed_when判断被控端数据没有发生改变，notify不通知了，所以handlers不执行

  • 强制执行handlers是在你的task执行失败的强制调用handlers
