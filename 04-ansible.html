Ansible 简介

  • Ansible是IT自动化管理工具，Python语言开发，通过SSH协议通信
  • 特点：容易学习、无需代理、操作灵活、模块丰富、安全性高、移植性强
  • 主要功能：批量执行远程命令、批量配置软件服务、编排改进IT任务

配置文件优先级

从高到低：$ANSIBLE_CONFIG（环境变量）> 当前目录 ansible.cfg > 安装目录 ansible.cfg > /etc/ansible/ansible.cfg

ad-hoc 执行过程

  1. 加载 ansible.cfg 配置文件
  2. 读取 hosts 主机清单，查找目标主机
  3. 加载对应模块
  4. 将模块生成 .py 文件，传输到被控端 $HOME/.ansible/tmp/
  5. 添加执行权限并执行
  6. 返回执行结果到主控端
  7. 清理被控端临时文件

主命令主机名称指定模块模块动作命令
ansibleall / 主机组 / IP-m（固定）模块名-a（固定）模块参数

Ansible-playbook（YAML 规范 / 变量系统 / Register / Loop / Handlers / Tags / Include / 异常处理）

▼

Playbook 简介

  • Playbook是yml语法编写的文本文件，由 play 和 task 组成
  • play：定义主机、主机组（用 hosts: 定义）
  • task：定义对主机或主机组的任务（用 tasks: 定义）
  • playbook 是 ad-hoc 的一种编排方式，可以持续运行，适合复杂任务

YAML 编写规范

1. 禁止使用Tab缩进，必须使用空格，且空格数必须是双数（2、4、6、8...）
2. 文件编码格式必须与操作系统匹配（默认utf-8）
3. 缩进：每个缩进由双数空格对应，通常2个空格代表一层
4. 冒号：除了以冒号结尾的，所有冒号后面必须有一个空格
5. 减号：减号代表项，减号后面必须有空格

Playbook 基础案例

# 在被控端创建文件
- hosts: webservers
  tasks:
    - name: 创建一个文件
      file: name=/opt/playceshi.txt state=touch

# 在不同被控端执行不同操作
- hosts: 192.168.10.129
  tasks:
    - name: 在129创建文件
      file: name=/opt/129.txt state=touch
- hosts: 192.168.10.130
  tasks:
    - name: 在130创建第一个文件
      file: name=/opt/130-1.txt state=touch
    - name: 在130创建第二个文件
      file: name=/opt/130-2.txt state=touch

# 执行playbook
ansible-playbook an-1.yml

变量系统

变量定义方式
方式说明适用场景
vars在playbook文件内头部定义当前文件使用
vars_files引用外部yml文件定义变量所有playbook共用
Inventory在/etc/ansible/hosts中针对主机/主机组定义特定主机数据
host_vars创建host_vars/目录，文件名=主机IP每台主机特有数据
group_vars创建group_vars/目录，文件名=主机组名整组通用数据
-e / --extra-vars执行playbook时命令行传参通用执行操作，优先级最高
变量优先级（从高到低）
1. playbook外部定义的变量（-e / --extra-vars）
2. vars_files 外部定义的变量文件
3. vars 在playbook文件内定义的变量
4. host_vars 针对主机定义的变量
5. Inventory /etc/ansible/hosts 针对个别主机的变量
6. group_vars 针对主机组定义的变量
7. Inventory /etc/ansible/hosts 针对主机组定义的变量

变量优先级验证实验
# 实验1：验证 vars_files > vars
# 同时存在 vars 和 vars_files 定义同名变量 baidu_url
- hosts: webservers
  vars:
    baidu_url: http://360.com
  vars_files: ./ceshi.yml      # ceshi.yml 中 baidu_url: http://www.baidu.com
  tasks:
    - name: "输出变量"
      debug:
        msg: "输出baidu的网址： {{ baidu_url }}"
# 结果：输出 http://www.baidu.com，说明 vars_files 覆盖了 vars

# 实验2：验证 vars > inventory
# inventory中定义了 uname=葫芦娃/果宝特攻，playbook中 vars 定义 uname: vars
- hosts: webservers
  vars:
    - uname: vars
  tasks:
    - name: "输出变量"
      debug:
        msg: "uname: {{ uname }}"
# 结果：两台主机都输出 "uname: vars"，说明 vars 覆盖了 inventory

# 实验3：验证 vars > host_vars
# host_vars/192.168.10.129 定义 hostname: 129，playbook 中 vars 定义 hostname: zhangsan
- hosts: webservers
  vars:
    - hostname: zhangsan
  tasks:
    - name: "输出host_vars目录下的变量"
      debug:
        msg: "{{ hostname }}"
# 结果：两台主机都输出 "zhangsan"，说明 vars 覆盖了 host_vars

# 实验4：验证 host_vars > inventory
# inventory 中 192.168.10.130 hostname=balabala
- hosts: webservers
  tasks:
    - name: "输出host_vars目录下的变量"
      debug:
        msg: "{{ hostname }}"
# 结果：129输出129，130输出130（host_vars），说明 host_vars 覆盖了 inventory

# 实验5：验证 -e 传值 > vars_files
# ceshi.yml 中定义 server: http，执行时 -e "server=firewalld"
- hosts: webservers
  vars_files: ./ceshi.yml
  tasks:
    - name: "输出执行时传入的值"
      debug:
        msg: "服务： {{ server }}"
# 执行：ansible-playbook an-8.yml -e "server=firewalld" -e "id=890"
# 结果：输出 "服务： firewalld"，说明 -e 传值优先级最高，覆盖了 vars_files

# 实验6：验证 group_vars > inventory
# inventory 中 192.168.10.130 hostage=8，[webservers:vars] hostage=9999
# group_vars/webservers 定义 hostage: 3年
# 结果：两台主机都输出 3年，说明 group_vars 覆盖了 inventory
# 再添加 [webservers:vars] hostage=9999，结果还是 3年，说明 group_vars 优先级高于 [webservers:vars]

Register - 注册变量

将任务执行结果保存到变量中，便于后续使用。

- hosts: webservers
  tasks:
    - name: "获取信息，赋值到register中"
      shell: ls /opt
      register: opt

    - name: "输出命令的执行结果"
      debug:
        msg:
          - "输出命令的执行结果：{{ opt.stdout_lines }}"

Register 常用属性

属性说明
changed变量是否改变了状态
stdout_lines保存的执行结果（最常用）
failed是否有错误
cmd执行的指令
start / end指令开始/结束时间
delta执行耗时

Loop 循环

# 基本循环 - 启动多个服务
- hosts: webservers
  tasks:
    - name: "启动服务：{{ item }}"
      service: name={{ item }} state=started
      loop:
        - httpd
        - mysql
        - firewalld

# 循环创建文件
- hosts: webservers
  tasks:
    - name: "循环创建三个文件"
      shell: chdir=/opt touch {{ item }}
      loop:
        - zhangsan.txt
        - lisi.txt
        - wangwu.txt

# JSON格式循环 - 创建用户组并添加用户
- hosts: webservers
  tasks:
    - name: "创建用户组：{{ item }}"
      group: name={{ item }} state=present
      loop:
        - gp1
        - gp2
        - gp3

    - name: "创建用户，设置附加组"
      user: name={{ item.uname }} groups={{ item.groups }} state=present
      loop:
        - { uname: "yh1", groups: "gp1" }
        - { uname: "yh2", groups: "gp2" }
        - { uname: "yh3", groups: "gp3" }

Handlers 和 Notify

Handlers是触发器，不能直接运行，需要被task的notify通知才执行。无论有多少task通知同一个handler，handler只在所有task执行完毕后执行一次。

- hosts: webservers
  tasks:
    - name: "安装http服务"
      yum: name=httpd state=latest

    - name: "下发配置文件"
      copy: src=/opt/httpd.conf dest=/etc/httpd/conf/httpd.conf
      notify:
        - restart_httpd      # 通知handler

    - name: "启动httpd服务"
      service: name=httpd state=started

  handlers:
    - name: restart_httpd       # 与notify名称对应
      service: name=httpd state=restarted

Tags 任务标签

# tags可以给task打标签，执行时指定或跳过
- hosts: webservers
  tasks:
    - name: "安装http服务"
      yum: name=httpd state=latest
      tags:
        - install_http
        - yum_http

    - name: "下发配置文件"
      copy: src=/opt/httpd.conf dest=/etc/httpd/conf/httpd.conf
      notify:
        - restart_httpd
      tags:
        - copy

    - name: "启动httpd服务"
      service: name=httpd state=started
      tags:
        - yum_http

  handlers:
    - name: restart_httpd
      service: name=httpd state=restarted

# 执行指定tag
ansible-playbook an-14.yml -t yum_http          # 只执行yum_http标签的任务
ansible-playbook an-14.yml --tag=yum_http         # 同上
ansible-playbook an-14.yml --skip-tags=yum_http   # 跳过yum_http标签的任务

Include 任务复用

# 定义公共任务文件 restart_http.yml（只定义任务，不含hosts）
- name: "restart_http"
  service: name=httpd state=restarted

# 在playbook中引用
- hosts: webservers
  tasks:
    - name: "任务一"
      shell: echo "任务一执行成功！！！！"
    - name: "重启http，调用yml文件完成"

Playbook 异常处理

# 1. ignore_errors - 忽略错误继续执行后续任务
- hosts: webservers
  tasks:
    - name: "错误任务"
      shell: asbdasdasdsd
      ignore_errors: yes        # 忽略错误，继续执行

    - name: "创建文件"
      file: path=/opt/123.txt state=touch

# 2. force_handlers - task失败时强制调用handlers
- hosts: webservers
  force_handlers: yes           # 强制执行handlers（定义在hosts下）
  tasks:
    - name: "创建文件"
      file: path=/opt/456.txt state=touch
      notify:
        - restart_httpd
    - name: "安装服务"
      yum: name=wwh state=latest
      ignore_errors: yes
  handlers:
    - name: restart_httpd
      service: name=httpd state=restarted

# 3. changed_when - 控制任务状态报告
# 重复创建文件，数据本身没有发生改变，但文件创建时间改变=被控端发生改变通知handlers
# 利用 changed_when 判断被控端数据没有发生改变，notify不通知了，所以handlers不执行
- hosts: webservers
  force_handlers: yes
  tasks:
    - name: "创建文件"
      file: path=/opt/456.txt state=touch
      notify:
        - restart_httpd
      changed_when: false       # 不报告changed状态

    - name: "安装服务"
      yum: name=wwh state=latest
      ignore_errors: yes
  handlers:
    - name: restart_httpd
      service: name=httpd state=restarted

# 4. 管理Nginx，具备检查功能（an-18.yml）
- hosts: webservers
  tasks:
    - name: "配置安装源"
      shell: rpm -Uvh https://nginx.org/packages/centos/7/noarch/RPMS/nginx-release-centos-7-0.el7.ngx.noarch.rpm
      tags:
        - uvh

    - name: "安装Nginx"
      yum: name=nginx state=present

    - name: "下发配置文件，更改Nginx默认端口号"
      template: src=/opt/default.conf dest=/etc/nginx/conf.d/default.conf
      notify:
        - restart_nginx

    - name: "检查服务状态"
      shell: /usr/sbin/nginx -t
      register: checknginx
      changed_when:
        - checknginx.stderr.find("successful")
        - false

    - name: "启动nginx"
      service: name=nginx state=started enabled=yes

  handlers:
    - name: restart_nginx
      service: name=nginx state=restarted

命令参数属性化

将模块的指令参数作为模块的属性，提高可读性。

# 紧凑写法
- name: "创建用户"
  user: name=hlxjg password="{{ '123456' | password_hash('sha512') }}"

# 属性化写法（推荐）
- name: "创建用户"
  user:
    name: hlxjg
    password: "{{ '123456' | password_hash('sha512') }}"

# 密码加密 - 使用password_hash过滤器
# ad-hoc方式
ansible webservers -m user -a 'name=ctds password={{ "123456" | password_hash("sha512") }}'

# 给用户提权
ansible webservers -m lineinfile -a 'path=/etc/sudoers insertafter="^root" line="ctds  ALL=(ALL) ALL"'

Delegate 委派

将原本需要在被控端执行的任务委派给其他主机运行。

委派给其他主机
# 在129机器执行任务，将其中一个任务委派给130执行
- hosts: 192.168.10.129
  tasks:
    - name: "添加一个host"
      shell: echo "www.baidu.com" >> /etc/hosts

    - name: "委派到130主机执行"
      shell: echo "www.wwh.com" >> /etc/hosts
      delegate_to: 192.168.10.130    # 委派到130执行

    - name: "再次追加数据"
      shell: echo "www.zhuijia.com" >> /etc/hosts
委派给主控机（run_once）
# 在主控机创建用户、生成密钥，然后下发到被控端
- hosts: webservers
  vars:
    - username: wwh
  tasks:
    - name: "委派给主控端，创建用户并生成密钥"
      user:
        name: "{{ username }}"
        generate_ssh_key: yes
        ssh_key_bits: 2048
        ssh_key_file: /opt/id_rsa
      register: userinfo
      delegate_to: localhost
      run_once: true                    # 只运行一次

    - name: "在被控端创建用户"
      user:
        name: "{{ username }}"
        password: "{{ 'abc123' | password_hash('sha512') }}"

    - name: "在被控端给用户提权"
      lineinfile:
        path: /etc/sudoers
        insertafter: "^root"
        line: "{{ username }} ALL=(ALL) ALL"

    - name: "被控端创建用户组"
      group:
        name: "{{ username }}"
        state: present
        system: yes

    - name: "用户家目录下创建.ssh"
      file:
        path: "/home/{{ username }}/.ssh"
        state: directory
        owner: "{{ username }}"
        group: "{{ username }}"
        mode: 0770

    - name: "下发主控端生成的密钥给被控端"
      copy:
        content: "{{ userinfo.ssh_public_key }}"
        dest: "/home/{{ username }}/.ssh/auto_keys"
        owner: "{{ username }}"
        group: "{{ username }}"
        mode: 0660

Jinja2 模板引擎

Jinja2是Python的全功能模板引擎，配合template模块实现配置文件中变量的解析替换。

copy vs template 对比
模块行为适用场景
copy直接复制文件内容，不做任何解析配置文件内容不需要改变
template解析文件中的变量，转换为对应值后复制每个设备需要不同数据
template 模块案例
# 1. 准备模板文件 tmp.conf（使用ansible系统变量）
vim tmp.conf
用户名称: {{ ansible_hostname }}
系统内存: {{ ansible_memtotal_mb }}MB
系统空闲内存: {{ ansible_memfree_mb }}MB

# 2. 编写playbook
- hosts: webservers
  tasks:
    - name: "使用copy模块（不会解析变量）"
      copy: src=/opt/tmp.conf dest=/opt

    - name: "使用template模块（会解析变量）"
      template: src=/opt/tmp.conf dest=/root

# copy结果：文件内容不变（原样输出{{ ansible_hostname }}等）
# template结果：变量被替换为实际值（如 localhost、972MB等）
Nginx 端口配置模板案例
# nginx配置文件 default.conf（使用变量）
server {
    listen       {{ my_port }};
    server_name  localhost;

    location / {
        root   /usr/share/nginx/html;
        index  index.html index.htm;
    }

    error_page   500 502 503 504  /50x.html;
    location = /50x.html {
        root   /usr/share/nginx/html;
    }
}

# playbook
- hosts: webservers
  vars:
    - my_port: 6666
  tasks:
    - name: "配置安装源"
      shell: rpm -Uvh https://nginx.org/packages/centos/7/noarch/RPMS/nginx-release-centos-7-0.el7.ngx.noarch.rpm
      tags: uvh
    - name: "安装Nginx"
      yum: name=nginx state=present
    - name: "下发配置文件，更改Nginx默认端口号"
      template: src=/opt/default.conf dest=/etc/nginx/conf.d/default.conf
      notify:
        - restart_nginx
    - name: "检查服务状态"
      shell: /usr/sbin/nginx -t
      register: checknginx
      changed_when:
        - checknginx.stderr.find("successful")
        - false
    - name: "启动nginx"
      service: name=nginx state=started enabled=yes
  handlers:
    - name: restart_nginx
      service: name=nginx state=restarted
