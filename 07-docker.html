Docker 基础

▼

1.1 Docker 介绍

  • 是一个开源、轻量级的虚拟化计算的容器引擎项目
  • 一个可以用于开发、发布和运行应用程序的开发平台
  • 基于 GO 语言开发，遵循 apache2.0 协议

1.2 Docker 的应用场景

  • web 应用的自动化打包和发布
  • 自动化测试和持续基础、发布
  • 在服务型的环境内部署和调整数据库或者其他的应用
  • 从头编译或者扩展现有 openshift 等等平台搭建自己的 PaaS 环境

1.3 Dcoker的架构

  • 镜像（image）：Docker 的镜像，就相当于 root 文件系统
  • 容器（container）：
    • 镜像和容器之间关系：镜像是静态的定义，容器就是镜像运行时的载体
    • 容器的管理：创建、启动、停止、删除、暂停等等
  • 仓库（repository）：存放镜像
  • 客户端：docker 客户端通过命令行或者其他工具使用 docker sdk 和 docker 的守护进程通信
  • 主机：一个物理或者虚拟的机器用于执行 docker 守护进程和容器

1.4 Docker 和传统虚拟机的区别

docker 误解：docker 是一个轻量级的虚拟机

（图：Docker 与传统虚拟机架构对比图）

  • 传统虚拟机
    • vmware 在宿主机的操作系统基础上，创建虚拟层、虚拟化操作系统、虚拟化的仓库等等，然后安装使用
  • docker 容器
    • 在宿主机操作系统上，创建 docker 引擎，在引擎的基础上安装应用

Docker 优势

  • docker 容器很快：启动或者停止都实现秒级完成
  • docker 容器对于系统资源需求很小，一个主机可以同时运行上千个 docker 容器
  • docker 通过类似 git 的操作方式方便用户的获取、发布、更新镜像，指令很简单
  • docker 通过 dockerfile 配置文件来支持更加灵活自动化创建和部署机制，通高工作效率
  • docker 保证应用性能的同时，尽可能减少系统开销
  • docker 利用 Linux 系统内的多种保护机制实现严格可靠的隔离
  • docker 引入安全选项和镜像签名机制，极大性提高 docker 的安全性
特点容器虚拟机
启动速度秒级分钟级
硬盘占用一般是 MB一般是 GB
性能接近原生弱于原生
系统支持量一个主机可以同时运行上千个 docker 容器一般就是几十个
  • docker build：编译 dockerfile 的命令
    • 根据的 docker file 创建一个可执行的 image
  • docker run：在有了可执行 image，就可以运行程序

Dockerfile 构建镜像

▼

8.1 Dockerfile 的常用命令

FROM

  • 指明构建镜像来自于那个基础镜像，如果你没有写 tag，那么默认还是 latest
  • 如果在构建的时候，不需要基础镜像，FROM scratch
FROM <镜像>:<tga>

MAINTAINER

  • 指明镜像的维护者，以及联系方式
  • 推荐是 LABEL
MAINTAINER <名字>
LABEL maintainer="名字"

RUN

  • 创建镜像的时候运作的 shell 命令
  • 比如：构建镜像的时候在 opt 目录下创建 a 目录
RUN mkdir -p /opt/a

ADD

  • 拷贝文件或者目录到镜像里面
  • ADD 在拷贝压缩包时，会自动解压
ADD src dest

COPY

  • 拷贝文件或者目录到镜像里面
  • COPY 在拷贝压缩包时，不会自动解压
COPY src dest

EXPOSE

  • 暴漏容器运行时监听端口给外部，可以指定端口监听 TCP 或者时 UDP
  • 默认 TCP
EXPOSE port

ENV

  • 设置容器的环境变量
ENV JAVA_HOME=/JDK

CMD

  • 启动容器时执行的 shell 命令
  • 注意：在 docker file 文件只能由一个 cmd 命令
    • 如果设置了多个 cmd 命令，只有最后一个会生效
  • 语法：
    • shell 格式：CMD 命令
    • exec 格式： CMD 可执行文件， 参数1……
    • 参数列表格式： CMD 参数1，参数2，参数3……
  • CMD 设置的命令会被 docker run 命令行指定的参数覆盖
CMD echo $JAVA_HOME
----------------------------------

docker run -it --name centos centos:7 echo "zhangsan"
#本身已经在docker file文件内容定义了执行的语句，但是如果在创建并且启动容器的时候
#重新赋予其他指令，那么在docker run 执行的语句设置的指令会覆盖CMD设置的指令
#像上面这个案例，在docker run 执行的时候，不会输出java_home的值，而是输出zhangsan

ENTRYPOINT

  • 指定一个容器启动的时候要运行的命令，和 CMD 应用都是指定容器在启动程序已经参数
    • ENTRYPOINT 不会被 docker run 命令行执行的指令参数覆盖
    • docker file 文件内容只能有一个 ENTRYPOINT 指令，如果设置多个，依旧只有最后一个生效
ENTRYPOINT echo $JAVA_HOME

WORKDIR

  • 指明在创建容器后，指定默认登录的目录
WORKDIR /dev

VOLUME

  • 指定容器挂载点到宿主机自动生成目录或者其他容器
  • 应用场景
    • 一般使用的场景需要持久化存储数据
    • 一般不会在 docker file 中使用
    • 常见的时 docker run 使用利用 -v 指定数据卷

8.2 编写 Dockerfile 文件

vim dockerfile
FROM centos:7
# 基于centos7镜像构建新的镜像

LABEL maintainer="zhangsan"
# 指定作者信息

WORKDIR /usr/local
#设置工作目录

ADD apache-tomcat-9.0.78.tar.gz /opt
ADD jdk-11.0.14_linux-x64_bin.tar.gz /opt
# 拷贝tar包到镜像中的opt目录并进行解压

RUN cd /opt && mv jdk-11.0.14 /usr/local/jdk11 && mv apache-tomcat-9.0.78 /usr/local/tomcat9
#移动文件重命名

EXPOSE 8080
#暴漏8080端口给外部

ENV JAVA_HOME=/usr/local/jdk11
ENV TOM_HOME=/usr/local/tomcat9
ENV PATH=PATH:{JAVA_HOME}/bin:${TOM_HOME}/bin
# 声明了环境变量

CMD startup.sh && tail -f ${TOM_HOME}/logs/catalina.out
#启动tomcat并查询日志

构建镜像时的写法：

  • 构建的时候，需要的包和 dockerfile 文件如果不在同一目录下，在构建的时候，需要指明存放相关包的路径
    • 例如下列语句，需要的包存放在 /root 目录下
    • docker build -f dockerfile -t whcentos:7 /root
  • 如果需要构建 dockerfile 和需要的包在统一目录下可采用下面的方式
    • docker build -f dockerfile -t whcentos:7 .
