目标：在看s集群内部部署mysql，验证集群是否可以正常通信

[root@master home]# vim mysql-rc.yml

apiVersion: v1    #定义apiserver的版本
kind: ReplicationController    #指定pod控制器
metadata:    #定义资源的元数据
  name: mysql    #指定创建程序的名字
spec:    #定义deployment的参数
  replicas: 1    #定义副本数量
  selector:    #定义标签
    app: mysql
  template:    #定义模板
    metadata:
      labels:
        app: mysql
    spec:
      containers:
      - name: mysql    #定义容器的名称
        image: mysql:8.0.21    #指定镜像文件
        ports:
        - containerPort: 3306    #定义容器的对外端口
        env:
        - name: MYSQL_ROOT_PASSWORD    #设置登录密码
          value: "123456"
  • 创建service
[root@master home]# kubectl  apply -f mysql-rc.yml

  • 编写文件，暴漏端口

apiVersion: v1
kind: Service
metadata:
   name: mysql
spec:
  type: NodePort
  ports:
    - port: 3306
      nodePort: 31000
  selector:
    app: mysql

一、介绍

  • K8S集群内，所有的内容都可以被称为资源，需要去操作资源进行管理K8S程序

    • 程序是部署在容器内

    • 由于pod是K8S最小管理单位，所有容器是部署在pod内

    • pod由pod控制器进行管理

    • 程序需要对外访问，需要使用service

    • 程序内的数据持久化由volumes

pod工作流程

  • 管理员要创建pod，kubuctl 向apiserver发起创建pod请求

  • apiserver将pod的信息存储一份到etcd

  • controller-manager 通过监听apiserver的watch接口发现pod信息的更新，然后 整合所有的资源，整合发给apiserver，在这个时候这个pod就是可以调度，apiserver将信息写入etcd

  • scheduler 通过监听apiserver的watchd接口，更新pod可以被调度，通过短发给pod分配合适的节点，把pod和对应节点的信息通知apiserver

    • 先预选再优选

  • apiserver在收到pod和对应节点的信息后，会将信息存储到etcd内

  • apiserver会将pod的信息发送给指定node节点上的kubelet

  • kubelet在接收到pod信息后，通知docker创建容器

  • docker在创建完成后返回信息给kubelet

  • kubelet把pod创建信息、运行信息等等返回给apiserver，再由apiserver将信息写入etcd

  • K8S 本质就是集群系统，用户是可以再集群内部署各种服务

    • 就是在K8S集群内运行一个个的容器，并且指定程序运行到容器

    • 由于pod是K8S最小管理单位，如果你想要K8S帮你管理容器，就只能把容器防止到pod内，；K8S是不会直接管理容器了，而是通过pod控制器管理pod

    • pod可以提供服务器，但是集群外部是不可以直接访问pod服务，k8s通过service资源实现这个功能

    • 需要时间pod内程序数据的持久化，K8S提供volumes存储系统

一、K8S 基础

Kubernetes 》 K8S

1.1 部署方式演变

传统部署》虚拟机部署》容器化部署

  • 传统部署：

    • 优点： 安装简单

    • 缺点：不能给程序定义边界

  • 虚拟化部署

    • 在一个物理机上运行多个虚拟机，给每一个虚拟机都提供单独的环境

    • 优点：程序环境之间是互不影响，安全性得到了保障

    • 缺点：增加了操作系统浪费了资源

  • 容器化部署

    • 优点：可以保证每一个容器都有自己的文件系统，cpu、内存、进程空间等等

      • 运行应用程序所需要的资源都能内容器包装

    • 问题

      • 如果一个容器关停了，怎么让另一个容器替补关停的容器呢？

      • 如果程序的访问量变多的时候，怎么横向扩展？

      • 当程序的访问量变少的时候，怎么回收呢？

为了解决这些容器管理的问题，使用相关的软件：

  • K8S：谷歌开源的容器编排工具

  • swarm：docker自己的容器编排工具

1.2 K8S 介绍
  • 基于容器技术的容器编排的解决方案
1.3 K8S的特点

  • K8S本质是一组服务器集群，可以在集群的每一个节点上运行特定的程序，对节点中的容器进行管理

  • 自我修复：一旦你某一个容器崩溃，能够在一秒左右启动新的容器

  • 弹性收缩：可以根据需要，自动对集群内正在运行的容器数量进行调整

  • 服务发现：服务可以通过自动发现的形式找到所依赖的服务

  • 负载均衡：如果一个服务启动了多个容器，能够自动实现请求的负载均衡

  • 版本回退：如果发现新发布的程序版本有问题，可以立即退会原来的版本

  • 存储编排：可以根据容器自身的需求自动创建存储卷

1.4 K8S集群组件
  • 搭建K8S集群的时候，服务器的数量尽量单数，方便选举master
1.4.1 K8S集群内主要是两种角色

  • master节点：控制节点

  • node节点：工作节点

1.4.2 master 节点

  • 集群的控制平台，复制集群的决策

  • master组件

    • kube-apiserver 端口：6443

      • 集群操作的唯一入口，接受用户输入命令，提供认证、授权、API注册、发现等等机制

    • kube-controller-manage 端口：10252

      • 负责维护集群的状态，比如：程序的部署安装、故障检查、自动扩展等等

    • kube-scheduler 端口：10251

      • 负责集群资源调度，按照预定的调度策略将pod调度到相应的node节点上

    • etcd 端口：2379 2380

      • 负责存储集群内各种资源对象的信息
1.4.3 node 节点

  • 集群的数据平台，负责为容器提供运行环境

  • node 组件

    • kubelet 端口：10250

      • 负责维护容器的生命周期，控制docker，来创建、更新、删除容器

    • kube-proxy

      • 负责提供集群内部的服务发现、负载均衡

    • docker

      • 容器引擎，负责本机容器创建和管理工作
